# Review: `docs/kernel-debug-skill-training-plan.md`

- **Reviewer:** ark-code (latest)
- **Review type:** Document review
- **Source file:** `docs/kernel-debug-skill-training-plan.md` (74 lines, 30-day plan)
- **Reference rules:** `README.md` positioning, `AGENTS.md` content/structure/verification rules
- **Scope of review:** Completeness for senior Linux kernel debugging skill, coverage of common problem types and symptoms, learning-order reasonableness, public-article-series kernel expertise, writing-style optimality, actionable improvements.

## Summary Assessment

The plan is a well-structured 30-day roadmap that follows the project's mandated "How / Why / Q/A" style and pairs mechanism knowledge with a daily artifact and verification line. It correctly frames debugging as the combination of mechanism, symptom, tool, and verification discipline, which matches `README.md` Section 5. The table form is appropriate for scanning, and the daily-article pattern at the top is a strong template.

However, the document falls short of "senior kernel debugging" completeness in a few important ways:

1. **Tooling coverage is shallow.** There is no dedicated day for `ftrace`/`trace-cmd`, `perf record/report/stat`, `bpftrace`, `kgdb`/QEMU gdb stub, `kdump`+`crash`, `printk`/dynamic-debug discipline, or `git bisect` regression isolation — all of which are explicitly listed in `README.md` Section 5.
2. **Several very common symptoms are absent.** Hard lockup / soft lockup / NMI watchdog, stack overflow (`CONFIG_VMAP_STACK`), uninitialized / out-of-bounds memory access (KASAN beyond UAF), `KFENCE`/`KCSAN`/`UBSAN`/`kmemleak`, module-unload races, power-management (suspend/resume) hangs, CPU/memory hotplug, memcg OOM, io_uring stalls, driver PIO/MMIO faults, TLB-shootdown / IPI storms.
3. **No `CONFIG_*` / boot-parameter binding.** Each day should name the concrete config and boot knobs that turn the relevant debug feature on (`CONFIG_DEBUG_INFO`, `CONFIG_KASAN`, `CONFIG_LOCKDEP`, `CONFIG_PROVE_LOCKING`, `CONFIG_DEBUG_ATOMIC_SLEEP`, `CONFIG_RCU_CPU_STALL_TIMEOUT`, `panic_on_oops`, `softlockup_panic=`, etc.). Without that, a learner cannot actually run the day.
4. **Verification column is too qualitative.** Many entries say "Can explain …" or "Can name …". Senior-level verification should be tied to reproducer inputs, expected counter values, or a specific observable signal.
5. **Sequencing has one real inversion and several late placements.** Day 1 asks the reader to annotate a panic before Day 2 establishes a reproducible lab; `perf`/`ftrace` instrumentation does not appear as a dedicated unit; "reproducer reduction" (Day 28) should be threaded through earlier crash and race days rather than introduced near the end.
6. **Public-article kernel expertise is uneven.** Real terminology is used, but the plan does not anchor days to kernel symbols (`__schedule`, `do_page_fault`, `__alloc_pages_slowpath`, `wait_event_*`, `lock_acquire`, `rcu_gp_*`) or to kernel version-specific behavior — required for credibility as a public series.

## Prioritized Action List

1. Add dedicated units for `ftrace`/`trace-cmd`, `perf`/`bpftrace`, `kdump`+`crash`, `kgdb`/QEMU gdb, `dynamic-debug`/`printk` discipline, and `git bisect`. (High — closes the README Section 5 coverage gap.)
2. Widen the memory-corruption cluster (Day 7+) to all KASAN/KFENCE/UBSAN/kmemleak symptoms, and add a separate unit for stack overflow and module-unload races. (High — common-symptom completeness.)
3. Add the lockup-table unit (hung/soft/hard/RCU) and the cgroup/memcg-OOM, io_uring-stall, PM/hotplug units. (High/medium — common symptoms in production.)
4. Bind `CONFIG_*` and boot knobs to every day. (Medium — turns the plan from concept into runnable.)
5. Replace soft "Can explain" verification with evidence-based acceptance criteria. (Medium — raises the bar to senior.)
6. Reorder so that the reproducible-lab day precedes the panic-reading day, and thread "reproducer reduction" through earlier days. (Medium — sequencing.)
7. Pin a reference kernel version at the top of the plan. (Info — version-specific behavior.)
8. Map outputs into `docs/04-debugging/` and `labs/<area>/` per `README.md`. (Info — alignment with repo structure.)

## Conclusion

The plan is a strong, style-compliant skeleton but is currently closer to a 30-topic syllabus than a senior-trainable roadmap. The structural shape (problem-first, mechanism-anchored, artifact-producing, verifiable) is correct and matches `AGENTS.md`. The main gaps are **tooling first-class-ness**, **sanitizer breadth**, **lockup/symptom coverage**, **config/boot-knob binding**, and **evidence-based verification language**. Filling those would lift the document from "good daily-article index" to a credible senior Linux kernel debugging curriculum consistent with `README.md` Section 5.
